# ゲーム企画書：Article Generator

## 1. 概要

本作は『クッキークリッカー』をベースとした**クリッカーゲーム**である。
記事を読まずにコンテンツを消費する現代の読者に対するアンチテーゼとして、**仏教観（輪廻転生、空、業、自我）の現代的解釈**をゲーム体験を通じて提示する。

---

## 2. ゲームシステム

### 2.1 基本サイクル

1.  **記事の生成**: 画面中央のキーボードとモニタをクリックして「記事」を生成する。
2.  **PV数**: 画面上部に表示。**常に3〜4の間をランダムに推移**し続ける（不変の虚無）。
3.  **施設の購入**: 蓄積した「記事」を消費して施設を開放し、自動生成（CPS: Content per Second）を強化する。

### 2.2 施設一覧

下表の順に高レベル・高効率な施設となる。

| # | 施設名 | 解説 |
| :---: | :--- | :--- |
| 0 | 人力（手動） | デフォルト。無強化状態で1クリックにつき1記事を生成。 |
| 1 | ゴーストライター | 低賃金で雇われた学生ライター。 |
| 2 | コピペ | Stack Overflow等からの無断転載。 |
| 3 | LLM | 内容は破綻しているが生成速度は速い。 |
| 4 | 自動執筆パイプライン | git pushに連動して記事を自動生成する。 |
| 5 | AGI | 汎用人工知能。人間による執筆を不要にする。 |
| 6 | 虚空からの抽出 | アカシックレコードへのハッキングによる情報取得。 |
| 7 | 猿のタイプライター | 無限の猿がシェイクスピアを書き上げる確率論的執筆。 |
| 8 | 情報の広義積分 | 情報を無限大に発散させ、宇宙を満たす。 |

### 2.3 悟りと転生（周回要素）

*   **転生の条件**: 累計「記事」数が**100兆**に達すると、「悟りに達しましたか？」という問いが表示される。
*   **業（カルマ）システム**: 初回の「悟り」により解放。2周目以降、特定の条件を満たすことで「得」を入手可能。
*   **得（徳）の用途**: 「有漏（うろ）の得」として、基礎CPSの向上などに使用（本家におけるヘブンリーチップス枠）。

## 3. 「得」システム

### 3.1 入手方法

「得」の入手方法は以下の2パターンを併用する。

#### 方法1：転生時の生産量に基づく計算

$$\text{得} = \sqrt[3]{\frac{\text{今回生産した記事総数}}{10^{15}}}$$

悟りに達した時点の周回で生産した記事総数から上記式で算出。
#### 方法2：座禅（放置プレイ）

ブラウザを最前面にした状態で、一切の操作を停止する「真の苦行」による入手。

*   **判定ロジック**:
    *   `mousemove`, `keydown`, `click`, `scroll` 等のイベントを監視し、入力時に `lastActionTime` を現在時刻に更新。
    *   **Page Visibility API** を監視し、タブが非表示（hidden）の間はカウントを停止。
*   **進行とリセット**:
    *   無操作状態（`現在時刻 - lastActionTime`）が1分経過すると「禅ゲージ」を表示。
    *   そのまま累計1時間が経過すると「得」を1入手。
    *   入力があった瞬間、画面に**『喝！』**という文字が表示され、ゲージは即座にリセットされる。


### 3.2「得」の用途（永続アップグレード）

| 強化名 | 効果 | 元ネタ・備考 |
| :--- | :--- | :--- |
| マルチスレッド輪廻 | 転生時の初期CpSが上昇する。 | 並列処理による効率的な生まれ変わり。 |
| サーバーレス涅槃 | オフライン時の生産効率が100%になる。 | 実体がなくても概念として記事が増える。 |
| 圧縮アルゴリズム「色即是空」 | 施設の価格上昇率が緩和される（1.15倍 → 1.14倍など）。 | データ量を「空」に近づけることでコストダウン。 |

### 数値処理に関する技術仕様（巨大数の取り扱い）

本プロジェクトでは、目標値が100兆（$10^{14}$）を超え、最終的には京、垓といった天文学的な数値に到達することを想定しています。JavaScript標準の数値型（Number）では約9,000兆（$2^{53}-1$）を超えると精度の問題が発生するため、以下の通りライブラリを導入して対応します。

- **採用ライブラリ**: `break_infinity.js`
- **導入理由**: 
    - クリッカーゲーム開発におけるデファクトスタンダードであり、軽量かつ高速。
    - 最大 $10^{308}$ までの数値を精度を維持したまま処理可能。
    - 指数表記や大規模な数値の加減算・比較を安全に行える。
- **実装方針**:
    - ゲーム内の通貨（記事数）、CpS（秒間生産量）、施設コスト等のすべての計算に本ライブラリを使用し、桁あふれや精度誤差による進行不能バグを防止する。
